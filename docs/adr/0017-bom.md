# ADR 0017: BOM and Manufacturing Package

## Status

Accepted for implementation.

## Context

Deficiency D4 in `docs/ca-sheet-metal-fixes.md` observes that `cadx` has no
bill-of-materials artifact and no part-metadata channel. The CLI exposes only
`init/run/inspect/render/evaluate/compare/loop`, and the registry records
published objects and features but nothing about *how to buy or make* a part:
vendor, material, thickness, finish, quantity, source/part number, unit cost, or
process.

Ordering laser-cut sheet metal (the SendCutSend workflow the spec targets) needs
a structured parts list. A shop quotes on material, thickness, finish, and
per-plate **area** plus fastener counts. Today an agent that converges a
multi-part assembly still has to hand-assemble that table from scattered run
artifacts, which is error-prone and non-deterministic.

The harness already derives the facts the BOM needs: ADR 0013/0016 record the
flat-pattern area on each DXF export, `spatial.json` carries each object's
bounding box, and ADR 0012's deduplicated features give hole counts. What is
missing is (a) a place for the agent to declare purchasing metadata and (b) a
command that joins it with the auto-derived geometry facts into a deterministic,
machine-readable package.

## Decision

Add a part-metadata channel and a `bom` subcommand.

**Authoring API.** `publish_part_meta(label, *, vendor=None, material=None,
thickness_mm=None, finish=None, qty=1, source_url=None, unit_cost=None,
part_number=None, process=None)`, re-exported from `cadx`, stores a record keyed
by `label` in a new `_PART_META` registry channel. The snapshot grows a
`part_meta` list, and `diagnostics["part_meta"]` carries it (all ten fields
always present so the record shape is fixed). The key is added to the worker
success envelope, the worker error envelope, and `runner._base_diagnostics`, so
it is always present like `exports`/`features`.

**`cadx bom <run_dir>`.** A new `src/cadx/bom.py` aggregator (read-only over a
completed run — it never re-executes the design) that, for each published object:

- derives `area_mm2`: the recorded flat-pattern area from the object's DXF export
  (the unfolded blank area, correct for both flat and bent parts), falling back
  to the largest planar-face area of the STEP solid, then to `null` with a
  warning when neither is available;
- derives `bbox_mm` from `spatial.json` and `hole_count` from the deduplicated
  `cylindrical_hole` features attributed to the object (unsourced features are
  attributed to the sole object when a run has only one);
- joins the object with its `part_meta` (objects without metadata still appear,
  with empty purchasing fields and `qty` defaulting to 1);
- writes a deterministic `bom.csv` and `bom.json`, rows sorted by
  `(vendor, label)`, grouped by vendor with per-vendor totals and a grand total
  of `qty`, `area_mm2`, and extended cost (`qty * unit_cost`).

**Determinism.** `bom.csv` is written through a fixed CSV dialect
(`lineterminator="\n"`, `QUOTE_MINIMAL`) with an explicit `newline=""` handle, a
fixed column order, and a stable row sort, so re-running yields byte-identical
output. Totals live in `bom.json` (under `vendors[]` and `totals`) rather than as
in-band CSV rows, keeping `bom.csv` a clean rectangular table. `bom.json` uses the
existing deterministic `write_json` (indent=2, sort_keys=True).

## Success Criteria

Written so the new tests fail before implementation and pass after:

- `test_part_meta_recorded_in_diagnostics`: `publish_part_meta` surfaces a record
  in `diagnostics["part_meta"]` (the key does not exist today).
- `test_bom_rows`: a plate with two holes plus `publish_part_meta` produces a BOM
  where the plate row carries the declared material/thickness/qty, a computed
  `area_mm2` within 0.01 mm² of the analytic flat area, and `hole_count == 2`;
  vendor totals equal the sum of member rows; the summary line agrees.
- `test_bom_deterministic`: running `cadx bom` twice yields a byte-identical
  `bom.csv` that round-trips through `csv.reader`.
- `test_bom_handles_part_without_meta_and_without_step`: a synthetic dict
  publication with no STEP yields empty `area_mm2` + a warning and a `qty` of 1,
  without crashing.
- Existing ADR 0001–0016 tests continue to pass; the contract gains only additive
  keys (`part_meta`, `bom.csv`/`bom.json`, and an `area_mm2` field on DXF records).

## Consequences

- The diagnostics contract gains a `part_meta` list and DXF export records gain an
  optional `area_mm2`; the run directory gains `bom.csv`/`bom.json`. No existing
  key is renamed or removed.
- `bom` is read-only and cheap to re-run, with a byte-for-byte determinism
  guarantee that is directly tested.
- Costing is intentionally minimal (`qty * unit_cost`); currency, tax, and freight
  are out of scope for downstream tooling that consumes `bom.json`.

## After Action Report

The red state failed as predicted: `from cadx import publish_part_meta` raised
`ImportError`, `diagnostics` had no `part_meta` key, and `cadx bom` was an
unrecognized subcommand.

An adversarial review of the diff confirmed the determinism guarantee (a live
`cmp` of two `bom.csv` renders) and the `part_meta`-in-all-three-envelopes
contract, and drove four fixes:

1. **`qty=0` was silently coerced to 1** (`int(meta.get("qty") or 1)`), re-adding a
   zeroed-out line item with cost. It now honors an explicit 0 and only defaults
   when `qty` is absent (`test_bom_qty_zero_preserved_and_orphan_meta_warns`).
2. **Sheet-metal area was wrong.** `publish_sheet_metal` stores the *folded* solid,
   so deriving area from its largest planar face returned a single flange, not the
   developed blank. The DXF exporters (flat, auto-flatten, and sheet-metal) now
   record the true `area_mm2` from the flat profile, and the BOM prefers it over
   the STEP-face fallback, so a bent part quotes on its unfolded area
   (`test_bom_sheet_metal_area_is_flat_pattern`: ~2153 mm², not the 1200 mm²
   base flange).
3. **Orphan `part_meta`** (metadata for a label that was never published) was
   silently dropped; it now surfaces a `bom_orphan_part_meta` warning so a typo'd
   label is not lost from the order.
4. **`totals.area_mm2`/`ext_cost`** were `int 0` for an empty sum; they are now
   coerced to `float` for a stable machine type.

The CSV intentionally omits in-band subtotal/total rows (the design sketch had
`__subtotal__`/`__total__` rows); totals live in `bom.json`'s `vendors[]` and
`totals`, which keeps `bom.csv` a clean rectangular table that imports without
filtering. The `_flat_area` STEP fallback remains for parts that have a STEP but
no DXF (non-prismatic solids); the full suite passed with 76 tests (69 prior + 7
new), no regressions.
